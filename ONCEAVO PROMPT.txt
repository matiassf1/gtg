════════════════════════════════════════════════════════════════
  OBJETIVO
════════════════════════════════════════════════════════════════

Construir la página completa de detalle de un restaurante para
el cliente: fotos, información, menú, reseñas, ubicación,
botón de reserva y favoritos.


════════════════════════════════════════════════════════════════
  1. CARPETAS CREADAS
════════════════════════════════════════════════════════════════

Se crearon los directorios necesarios para las nuevas rutas:

  app/api/cliente/reservas/          ← API para crear reservas
  app/api/cliente/resenas/           ← API para crear reseñas
  app/api/cliente/favoritos/         ← API para toggle favorito
  app/(cliente)/cliente/restaurante/[id]/  ← Página de detalle


════════════════════════════════════════════════════════════════
  2. API — POST /api/cliente/reservas
════════════════════════════════════════════════════════════════

ARCHIVO: app/api/cliente/reservas/route.ts

QUÉ HACE:
  Recibe una solicitud de reserva del cliente y la guarda en la DB.

FLUJO:
  1. Verifica sesión y que el usuario tenga rol CLIENTE
  2. Busca el registro Client vinculado al usuario (necesario para
     el campo clientId de la Reservation)
  3. Recibe del body: restaurantId, date (YYYY-MM-DD), time (HH:MM),
     guests (número), notes (texto opcional)
  4. Combina date + time en un solo DateTime:
     new Date(`${date}T${time}:00`)
  5. Valida que la fecha resultante sea en el futuro (no pasada)
  6. Crea la Reservation con status PENDIENTE
  7. Devuelve la reserva creada con status 201

VALIDACIONES:
  - Todos los campos requeridos deben estar presentes
  - El DateTime combinado debe ser válido (isNaN check)
  - La fecha debe ser estrictamente futura


════════════════════════════════════════════════════════════════
  3. API — POST /api/cliente/resenas
════════════════════════════════════════════════════════════════

ARCHIVO: app/api/cliente/resenas/route.ts

QUÉ HACE:
  Crea una nueva reseña de un cliente sobre un restaurante y
  recalcula automáticamente el promedio de valoración.

FLUJO:
  1. Verifica sesión y rol CLIENTE
  2. Busca el registro Client del usuario
  3. Verifica que el cliente NO haya reseñado ya ese restaurante
     (una reseña por cliente por restaurante)
  4. Crea el Review con rating (1-5) y comment opcional
  5. Recalcula el averageRating del restaurante:
     - Trae todos los ratings de ese restaurante
     - Calcula el promedio
     - Redondea a 1 decimal (Math.round(avg * 10) / 10)
     - Actualiza el campo averageRating en Restaurant
  6. Devuelve la reseña creada con datos del cliente (para
     mostrarla inmediatamente en la UI sin recargar)

MANEJO DE DUPLICADOS:
  Si el cliente ya tiene una reseña para ese restaurante,
  devuelve HTTP 409 Conflict con mensaje descriptivo.
  El frontend muestra ese error al usuario.


════════════════════════════════════════════════════════════════
  4. API — POST /api/cliente/favoritos
════════════════════════════════════════════════════════════════

ARCHIVO: app/api/cliente/favoritos/route.ts

QUÉ HACE:
  Toggle de favorito: si el restaurante ya está en favoritos
  lo quita, si no está lo agrega. Una sola ruta para ambas
  operaciones (no hay DELETE separado).

FLUJO:
  1. Verifica sesión y rol CLIENTE
  2. Busca el Client con su lista de favorites incluida
  3. Recibe restaurantId del body
  4. Verifica si el restaurante ya está en la lista de favoritos
  5. Usa Prisma relation para connect/disconnect:
     alreadyFavorited → { disconnect: { id: restaurantId } }
     no favorited     → { connect:    { id: restaurantId } }
  6. Devuelve { favorited: boolean } indicando el nuevo estado

USO EN FRONTEND:
  El cliente llama a este endpoint y actualiza su estado local
  según el booleano recibido, sin necesidad de recargar la página.


════════════════════════════════════════════════════════════════
  5. COMPONENTE PRINCIPAL — restaurante-detail.tsx
════════════════════════════════════════════════════════════════

ARCHIVO: components/cliente/restaurante-detail.tsx

Es un Client Component que contiene toda la interactividad de
la página. Está organizado en sub-componentes bien definidos.

────────────────────────────────────────────────────────────────
  5a. SUB-COMPONENTE: Stars
────────────────────────────────────────────────────────────────

  Renderiza 1-5 estrellas con tres tamaños (sm, md, lg).
  Las estrellas activas son amarillas (fill-yellow-400),
  las inactivas son grises (fill-muted). No es interactivo
  (solo display). El componente interactivo de estrellas
  está en el modal de reseñas.


────────────────────────────────────────────────────────────────
  5b. SUB-COMPONENTE: PhotoCarousel
────────────────────────────────────────────────────────────────

  Carrusel de fotos del restaurante.

  CUANDO HAY FOTOS:
    - Muestra la foto activa con next/image (fill + object-cover)
    - Botones prev/next circulares con fondo negro semitransparente
    - Overlay degradado de negro en la parte inferior
    - Puntos de navegación en el fondo: el activo es una pastilla
      ancha, los inactivos son círculos pequeños (como iOS)
    - Contador "1 / N" en esquina superior derecha
    - Transición de opacidad entre fotos

  CUANDO NO HAY FOTOS:
    - Muestra un gradiente oscuro con la inicial del restaurante
      en letras gigantes translúcidas como placeholder

  ANCHO: ocupa todo el ancho del contenedor con -mx-4 para
  "escapar" del padding del layout y verse full-width.


────────────────────────────────────────────────────────────────
  5c. SUB-COMPONENTE: MenuTab
────────────────────────────────────────────────────────────────

  Muestra el menú del restaurante agrupado por categorías.

  AGRUPACIÓN:
    - Reduce el array de menuItems en un Record<category, items[]>
    - Ordena las categorías alfabéticamente
    - Muestra cada categoría como encabezado con líneas decorativas
      a los costados (estilo ─── CATEGORÍA ───)

  CADA ITEM:
    - Imagen opcional (si el plato tiene foto)
    - Nombre del plato (tachado si available=false)
    - Descripción breve (limitada a 2 líneas)
    - Precio formateado como moneda argentina (toLocaleString)
    - Badge "No disponible" si available=false
    - Opacidad reducida para platos no disponibles

  ESTADO VACÍO:
    - Ícono + mensaje cuando el restaurante no tiene platos cargados


────────────────────────────────────────────────────────────────
  5d. SUB-COMPONENTE: LeaveReviewModal
────────────────────────────────────────────────────────────────

  Modal para que el cliente deje una reseña del restaurante.

  SELECTOR DE ESTRELLAS:
    - 5 estrellas interactivas grandes (w-8 h-8)
    - Hover: resalta hasta esa estrella y escala la estrella (scale-125)
    - Click: fija el rating
    - Texto descriptivo debajo del rating seleccionado:
      1 → "Muy malo", 2 → "Malo", 3 → "Regular",
      4 → "Bueno", 5 → "Excelente"

  TEXTAREA:
    - Campo opcional para comentario extendido
    - Placeholder con ejemplos

  VALIDACIÓN:
    - No se puede publicar sin seleccionar estrellas
    - Si ya dejó una reseña: muestra error 409 como toast
    - Botón deshabilitado si rating === 0

  AL PUBLICAR:
    - Llama a POST /api/cliente/resenas
    - En éxito: agrega la reseña al array local (sin recargar página),
      cierra el modal, muestra toast de confirmación


────────────────────────────────────────────────────────────────
  5e. SUB-COMPONENTE: ReviewsTab
────────────────────────────────────────────────────────────────

  Muestra la lista de reseñas con opciones de ordenamiento.

  CONTROLES:
    - Select de orden: Más recientes / Mejor valoradas / Peor valoradas
    - Contador de reseñas
    - Botón "Dejar reseña" (solo si isLoggedIn y !hasReviewed)
    - Badge "Ya dejaste una reseña" con check verde (si reviewed)

  CADA RESEÑA:
    - Avatar del cliente con iniciales como fallback
    - Nombre + fecha formateada
    - Estrellas del rating
    - Texto del comentario
    - Respuesta del restaurante (si existe) con borde izquierdo verde
      y label "Respuesta del restaurante"

  ORDENAMIENTO:
    - Se hace client-side con .sort() sobre una copia del array
    - "recent": por createdAt descendente
    - "best": por rating descendente
    - "worst": por rating ascendente

  ESTADO VACÍO:
    - Ícono + mensaje + invitación a ser el primero en reseñar


────────────────────────────────────────────────────────────────
  5f. SUB-COMPONENTE: MapTab
────────────────────────────────────────────────────────────────

  Muestra la ubicación del restaurante en un mapa.

  IMPLEMENTACIÓN:
    - iframe embebido de Google Maps (no requiere API key)
    - URL: maps.google.com/maps?q=DIRECCIÓN&output=embed&hl=es
    - La dirección se construye como: "address, city, Argentina"
    - El nombre de la dirección también se muestra en texto encima
      del mapa con ícono MapPin

  LIMITACIÓN:
    Google Maps basic embed funciona sin API key pero muestra
    una banda con "Ver mapa más grande". Es suficiente para
    orientar al usuario sobre la ubicación.


────────────────────────────────────────────────────────────────
  5g. SUB-COMPONENTE: HoursAccordion
────────────────────────────────────────────────────────────────

  Acordeón para los horarios del restaurante dentro del sidebar.

  ESTADO COLAPSADO:
    - Muestra solo el horario del día actual
    - Si hoy está cerrado: texto rojo "Cerrado hoy"
    - Si está abierto: "HH:MM – HH:MM"
    - Ícono ChevronDown que rota 180° al abrir

  ESTADO EXPANDIDO:
    - Lista los 7 días con sus horarios
    - El día actual se resalta en verde (text-primary font-semibold)
    - Días cerrados muestran "Cerrado" en lugar de horario

  DETECCIÓN DEL DÍA ACTUAL:
    - Usa new Date().getDay() (0=domingo, 1=lunes, ..., 6=sábado)
    - Mapea al key del objeto openingHours (lunes, martes, etc.)


────────────────────────────────────────────────────────────────
  5h. SUB-COMPONENTE: ReservaModal
────────────────────────────────────────────────────────────────

  Modal completo para hacer una reserva en el restaurante.

  CAMPOS:
    a) FECHA:
       - Input type="date" con min=hoy (no se puede reservar en el pasado)
       - Formato YYYY-MM-DD que se combina con la hora en el backend

    b) HORA:
       - Select con slots de 30 minutos desde las 12:00 hasta las 22:30
       - 23 opciones en total
       - El array TIME_SLOTS se genera dinámicamente con Array.from

    c) PERSONAS (contador +/-):
       - No es un input número sino dos botones y un display
       - Botón − (mínimo 1 persona)
       - Número actual en el centro
       - Botón + (máximo 20 personas)
       - Texto "persona/personas" se ajusta en singular/plural

    d) NOTAS (opcional):
       - Textarea con placeholder de ejemplos
       - Alergias, celebración especial, preferencia de mesa

  ESTADO DE ÉXITO:
    - Después de una reserva exitosa, en lugar del formulario
      aparece una pantalla de confirmación con:
      · Círculo verde con check
      · Mensaje explicando que el restaurante confirmará
      · Botón "Cerrar"
    - Al cerrar se resetea el formulario y el estado

  AL SUBMITTEAR:
    - Llama a POST /api/cliente/reservas
    - Maneja el error del backend mostrándolo como toast


────────────────────────────────────────────────────────────────
  5i. SUB-COMPONENTE: Sidebar
────────────────────────────────────────────────────────────────

  Panel lateral sticky con las acciones principales del usuario.

  BOTÓN RESERVAR:
    - Verde, grande (h-12), ocupa todo el ancho del sidebar
    - Abre el ReservaModal al hacer clic
    - Si isLoggedIn=false: deshabilitado

  BOTÓN FAVORITO:
    - Toggle: corazón relleno rojo = en favoritos, vacío = no está
    - Llama a POST /api/cliente/favoritos
    - Actualización optimista: el estado cambia inmediatamente
    - Toast de confirmación en ambos estados

  CARD DE PROMOCIÓN ACTIVA (si existe):
    - Fondo verde muy suave con borde verde sutil
    - Label "PROMOCIÓN ACTIVA" con ícono Tag
    - Título de la promo
    - Pill con el tipo (Descuento 20%, 2×1, etc.)
    - Fecha "hasta DD MMM"

  INFO RÁPIDA:
    - Teléfono (link tel:// para llamar directo en mobile)
    - Dirección con ícono
    - Acordeón de horarios (HoursAccordion)

  STICKY:
    - top calculado como: header (3.5rem) + padding del layout (1.5rem)
    - Solo se ve como columna en desktop (lg:flex-row)
    - En mobile, el sidebar va debajo del contenido principal


────────────────────────────────────────────────────────────────
  5j. COMPONENTE RAÍZ: RestauranteDetail
────────────────────────────────────────────────────────────────

  Orquesta todos los sub-componentes.

  LAYOUT:
    [Carousel full-width]
    [flex-col en mobile / flex-row en lg+]
      [Main content (flex-1)]
        - Header: badges de categoría + precio + promo, nombre, rating
        - Descripción del restaurante
        - Tabs: Menú | Reseñas | Ubicación
        - Contenido del tab activo
      [Sidebar (w-72, sticky)]

  TABS:
    - Estado local con useState<Tab>
    - Renderizado condicional según tab activo
    - Borde inferior activo en verde


════════════════════════════════════════════════════════════════
  6. PÁGINA SERVIDOR — /cliente/restaurante/[id]
════════════════════════════════════════════════════════════════

ARCHIVO: app/(cliente)/cliente/restaurante/[id]/page.tsx

QUÉ HACE:
  Server Component que junta todos los datos necesarios para
  la página y los pasa al componente cliente.

QUERIES A PRISMA:
  1. restaurant con include completo:
     - menuItems: ordenados por category ASC, order ASC
     - reviews: con client.user (nombre, email, imagen)
       ordenadas por createdAt DESC
     - promotions: filtradas activas (active=true, dentro del
       período validFrom/validUntil respecto a "now")

  2. client del usuario actual con:
     - favorites donde id = restaurantId (para saber si ya
       lo tiene en favoritos sin traer todos los favoritos)

  3. Review count para saber si ya reseñó:
     prisma.review.count({ where: { restaurantId, clientId } })

MANEJO DE NOT FOUND:
  Si el restaurante no existe: notFound() → página 404 de Next.js

SERIALIZACIÓN:
  Se convierten todos los campos DateTime a ISO string antes
  de pasar al Client Component (Next.js no puede pasar objetos
  Date entre Server y Client Components directamente).

PROPS PASADAS AL CLIENTE:
  - restaurant: datos completos serializados
  - isFavorited: boolean
  - hasReviewed: boolean
  - isLoggedIn: boolean (si existe registro Client)

════════════════════════════════════════════════════════════════
  FLUJO COMPLETO DE UNA RESERVA
════════════════════════════════════════════════════════════════

  1. Cliente entra a /cliente/restaurante/abc123
  2. Server Component fetcha el restaurante con su info completa
  3. Se renderiza la página con sidebar sticky visible
  4. Cliente hace clic en "Reservar mesa" (sidebar)
  5. Se abre el ReservaModal
  6. Elige fecha, hora, personas y agrega notas opcionales
  7. Click en "Confirmar reserva"
  8. POST /api/cliente/reservas con { restaurantId, date, time, guests, notes }
  9. Backend combina date + time → DateTime, crea Reservation PENDIENTE
  10. Modal muestra pantalla de éxito con check verde
  11. La reserva queda en la DB esperando confirmación del restaurante
  12. El restaurante la verá en /empresa/reservas en el tab "Hoy" o "Pendientes"


════════════════════════════════════════════════════════════════
  FLUJO COMPLETO DE UNA RESEÑA
════════════════════════════════════════════════════════════════

  1. Cliente está en la tab "Reseñas" del restaurante
  2. Ve el botón "Dejar reseña" (si no reseñó antes)
  3. Click → se abre LeaveReviewModal
  4. Selecciona estrellas (hover interactivo + texto descriptivo)
  5. Opcionalmente escribe un comentario
  6. Click en "Publicar reseña"
  7. POST /api/cliente/resenas con { restaurantId, rating, comment }
  8. Backend verifica que no sea duplicado
  9. Crea el Review, recalcula averageRating del restaurante
  10. Devuelve la reseña con datos del cliente
  11. La reseña aparece inmediatamente en la lista (sin recargar)
  12. El botón "Dejar reseña" desaparece, aparece "Ya dejaste una reseña"
  13. El restaurante la verá en /empresa/resenas y puede responderla
